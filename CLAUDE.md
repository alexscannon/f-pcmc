You are implementing F-PCMC, a frozen-encoder online unsupervised continual learning system. 
This file governs how you work in this repository across all sessions.

## Non-negotiable rules

- **One task per branch.** Branch name `task/TNN-short-name`. Do not mix tasks in one branch.
- **A task is done only when its listed tests pass AND `pytest` (full suite, including all prior tasks' tests) is green.** Never weaken, skip, mark xfail, or delete an existing test to make progress. If an existing test seems wrong, stop and report your reasoning instead of changing it.
- **Gate discipline.** Three tasks end in hard gates:
  - T6 `test_m1_gate` — LTM-only routing must match batch knn_vmf AUROC ±0.01
  - T11 `test_golden_stream_end_to_end` — executable spec of the full system (**GREEN as of 2026-07-11**; it reopened T9/T10 and found two real mechanism defects — see the T7–T11 bullets below. Keep it green: a later task that reddens it has broken a mechanism, not the gate.)
  - T14 `test_v1_regression_pin` — ported v1 must reproduce pinned numbers A red gate means the implementation is wrong, not the gate. Do not proceed past a red gate. Fix the implementation ("fix the port, never the pin"), or stop and report if you believe the pin itself is wrong.
  - **The golden gate's rate clauses are q-linked** (owner amendments 2026-07-11). FR-5.1 puts τ at the q-th percentile of a concept's *own* LOO scores, so ~(1 − q/100) of a concept's own in-distribution arrivals fall beyond its own τ **by construction** (measured: 0.0569 at q=95). That is a hard ceiling of `q/100` on any "routed to the right concept at tier 1" rate and a hard floor of `1 − q/100` on any "still unknown" rate — no implementation can beat it. The three gate clauses that measure such a rate therefore derive their bounds from `config.tau_percentile_q` ± a 3σ binomial band, **computed in the test, never hardcoded**. If you change `tau_percentile_q`, they re-derive themselves; if you write a new rate clause, respect the same ceiling.
- **No ground-truth leakage.** Nothing under `fpcmc/` may import from `eval/`'s ground-truth mapping or receive labels at runtime. The invariant tests enforce this; design accordingly.
- **No learning.** No torch autograd, optimizers, or model forward passes anywhere in `fpcmc/`. Embeddings are precomputed inputs.
- **Reference code is read-only.** `reference/stam` and `reference/pcmc` are populated git submodules (STAM, PCMC/`upl-benchmark`), pinned at fixed commits — do not `git submodule update --remote` them or edit their contents. You may read them (and the source papers at `research_papers/`) to understand conventions, but nothing in `fpcmc/`, `eval/`, or `tests/` may import from `reference/`. `test_no_reference_imports` enforces this.
- **`lib/` is a frozen vendored snapshot, read-only.** Verbatim byte-identical copies from the source project, blob hashes recorded in `lib/PROVENANCE.md`. Never edit files under `lib/`; extract/adapt the math at the consuming site (`fpcmc/`, `eval/`) with citation comments pointing back to the lib file.
- **Determinism everywhere.** All randomness flows through `fpcmc/rng.py::make_rng(seed, stream="")` (named substream per module), seeded from config. No module-level RNG, no unseeded library calls (UMAP must receive `random_state`). `test_byte_determinism` and per-module determinism tests enforce this.
- **The golden stream fixture AND its run config are frozen.** `tests/fixtures/golden_stream.npz` + the pinned sha256 in `tests/fixtures/golden_stream.py` are gate-input data for T8/T11 — never regenerate or re-pin without owner approval. As built (owner-approved 2026-07-10) it carries 25 one-off distractor outliers beyond the TASKS-enumerated composition. **`configs/golden_run.yaml` is owner-frozen too** (2026-07-11, at `stm_capacity=25`), on the same footing: it is gate *input*, not a tuning knob. A red golden gate is answered in `fpcmc/` — never by editing a value in that file.
- **Specs are immutable.** Never modify anything under `docs/` or `tests/reference_numbers.yaml`. Propose spec changes in your report instead. Standing exception: **appending the completed task's entry to `docs/CHANGES.md`** (owner-directed changelog, 2026-07-10). Any other `docs/` edit happens only on explicit owner instruction in-session (precedent: the dated 2026-07-10 as-built annotations in PRD §3/§9 and TASKS T0/T1/T8/T11/T12; the 2026-07-11 T6 pin re-source and the T10/T11 amendments), and is recorded in `docs/CHANGES.md`.

## Per-task workflow

1. Read the task's entry in the task plan and every PRD section it references. Read the tests it requires before writing implementation code.
2. Post a short plan (files to create/modify, test list, open questions). If any open question is a design decision not answered by the PRD, STOP and ask before implementing.
3. Write the task's tests first where practical (they are specified in the task plan); then implement until green.
4. Run: `pytest -m "not slow"` on every meaningful change; `pytest` (full, including slow/integration) before declaring the task done.
5. Commit in small logical units. Final commit message: `TNN: <summary> — all tests green (<N> passed)`.
6. Append the completed task's entry to `docs/CHANGES.md` (what/why/commits/tests/approved deviations — see the T0 entry for the format).
7. End the session with a report: what was built, test counts, runtime of integration tests vs NFR-1 budgets, any deviations proposed (never silently applied), and anything the next task should know.

## Stop conditions (halt and report rather than improvise)

- A gate test fails and the cause is not a clear implementation bug.
- A real-data schema check fails (`test_real_pool_schemas`).
- A pinned reference number appears unreachable within tolerance.
- The PRD is ambiguous on a decision that changes behavior.
- Any action would require editing `docs/`, `tests/reference_numbers.yaml`, or code under `reference/`.

Owner-approved deviations baked in (docs/CHANGES.md T2; dated notes at PRD FR-1/FR-4.3/FR-5): `Concept` carries per-sub-scorer thresholds `tau` (knn_ref) + `tau_vmf` (vmf); margins are `(τ−s)/|τ|` with lexicographic concept_id tie-break; the composed `knn_vmf` scalar score is its knn_ref sub-score (so composed acceptance is not derivable from the scalar). - **`Concept` dynamics are as-built at T3** (owner-approved decisions, docs/CHANGES.md T3): cached κ self-maintained by `add_observation` on every ref_set-changing observation (T4's lazy ≥25% trigger governs τ/τ_vmf only, never κ); the seeding embedding counts in `ref_count_seen` but not `match_count`/`match_windows` (maturity/θ counts are post-seed matches; `last_matched_at = created_at` at seed, so LRU is defined from birth); each concept carries its own reservoir Generator (per-concept named substream) plus `window_W`/`k_max`/`alpha_ema` fixed at construction, so `add_observation(z, step)` takes no per-call config; `provenance ∈ {"initial", "seeded", "promoted"}` (T8 flips "seeded"→"promoted"; FR-8.3's never-merge rule reads "initial" only); `concept_id` mutation raises; a Concept owns its arrays (the reservoir replaces rows in place — never rely on aliasing arrays passed in). The reservoir draw discipline (exactly one `rng.random(2)` pair per post-fill observation) is load-bearing for `test_reservoir_uniformity`'s exact vectorized replay — changing it requires updating that test's simulator in the same commit.
- **Thresholds are as-built at T4** (owner-approved decisions, docs/CHANGES.md T4): `fpcmc/thresholds.py` is the whole FR-5 stack, per sub-scorer (knn_ref → `tau`, vmf → `tau_vmf`). vmf LOO scores members against the cached (centroid, κ) — no per-member re-fit; knn_ref LOO masks the self-distance to `+inf`. Percentiles are `np.percentile` with the default linear interpolation, mirrored from `lib/.../knn_vmf.py::_calibrate_tau` — don't change the method. `recompute_thresholds(concept, config, prior)` is status-sensitive (LTM → pure FR-5.1 percentile; STM → FR-5.2 shrinkage; below the floors it holds the prior: τ at n=1, τ_vmf below `n_vmf_min`); promotion must instead use `recompute_on_promotion(concept, config)` (pure FR-5.1 for both taus, status-independent). The FR-5.1 lazy trigger is check-on-call: `Concept.refset_changes_since_tau` counts actual ref_set mutations, `maybe_recompute` fires at ≥ 25% of the current ref_set size, and every recompute resets the counter — the T5 store calls it after every assignment's `add_observation` (the sole call site). `GlobalPrior` (FR-5.3) is a frozen per-sub-scorer pair from `compute_global_prior(concepts, config)`, computed once at init. Anything that replaces a ref_set outside `add_observation` (T9 merges) must recompute κ itself before recomputing τ.
- **The ConceptStore/routing core is as-built at T5** (owner-approved decisions, docs/CHANGES.md T5): `fpcmc/concepts.py::ConceptStore(config, prior, concepts=(), *, vectorized=True)`; `route(z, step) -> RoutingResult` is the exact FR-9 cascade — tier 1 LTM ∪ tier1-eligible STM (`ConceptStore._tier1_stm`; **see the T11 bullet — eligibility is maturity AND cohesion, not maturity alone**), tier 2 the remaining STM (prediction "unknown"), tier 3 seed (prediction "unknown", score/margin NaN, via None). The two STM sets partition the registry via that single predicate, so the cascade stays total. `RoutingResult.score` is the winning ScoreDetail scalar (composed = knn_ref sub-score) plus `via`/`fallback` for the event log/A5. Tier membership is a live view over `status`/`match_count` — promotion-aware routing is just status flip + `recompute_on_promotion`, effective on the next call. Concept ids are store-allocated zero-padded `ltm_{:03d}`/`stm_{:04d}`, unique run-wide, never reused (`register` burns/advances past external ids; allocator overflow raises — id ordering feeds the FR-4.3 tie-break). The batch scoring path must stay bitwise-identical to the frozen scorers: per-concept `ref_set @ z` + vectorized composition/selection — never a stacked cross-concept GEMV (measured 1-ulp divergence; `test_vectorized_matches_loop` is the binding guard, `vectorized=False` is its reference loop). Invariant 5 is structural and AST-enforced: within `ConceptStore` only `__init__`/`_assign`/`_seed` may reference the prior. `route()` mutates the store — read-only scoring sweeps (T6 M1 gate) go through the scorer directly.
- **STM lifecycle, promotion and merging are as-built at T7–T10** (owner-approved, docs/CHANGES.md T7–T10). Eviction (FR-3.1) is a `ConceptStore` method draining at the sole STM growth site (`_seed`); victim = min `last_matched_at`, ties to older `created_at`, then smallest id; LTM is exempt, mature-but-unpromoted STM is not; evicted ids stay burned. `fpcmc/memory.py` holds `PromotionEvaluator` (four FR-7 criteria, all always evaluated; atomic promotion = status flip + provenance "seeded"→"promoted" + `recompute_on_promotion`) and `MergeSweeper` (FR-8.1 STM↔STM two-condition rule → FR-8.2 STM→LTM folds → FR-8.3 promoted-only LTM↔LTM, each phase to fixpoint with merges live). FR-8.1 merges union history; **folds move ref_set + `ref_count_seen` ONLY** (a fold is not a match event). `fpcmc/residual.py` (FR-6) shares the T9 sweeper so the run has ONE merge log/lineage map, and creates **no concept ids** — clusters drive merges of existing immature candidates; pool rows are density context only and never join a ref_set.
- **T11 closed the golden gate, and four rules keep it closed. Do not weaken them without re-running the gate.**
  1. **Tier-1 eligibility requires cohesion.** An STM candidate joins tier 1 only if `match_count >= n_mature` **AND** `cohesion >= min_cohesion` — a candidate too incoherent to ever be promoted is too incoherent to answer for a class. Without this, a candidate that accumulates the τ-tail rejects of many classes earns a τ several times looser than any well-calibrated concept's, and since the FR-4.3 margin is `(τ−s)/|τ|` a loose τ *raises* its margin for every query, so it outbids the correct LTM and captures its traffic (measured: cohesion 0.21, τ 3.4× the mean LTM τ, 47 known arrivals stolen in 84 steps) while being permanently unpromotable, unfoldable and beyond FR-6's reach.
  2. **`cohesion()` has ONE definition**, in `fpcmc/concepts.py` (a pure function of a ref_set); `fpcmc.memory` re-exports it. FR-7 promotion and the tier-1 gate must never drift apart. It is cached on `Concept` behind a lazy property invalidated on every ref_set mutation (`__setattr__` catches wholesale replacement — merges; `add_observation` catches in-place reservoir replacement).
  3. **Both merge guards are load-bearing.** FR-8.2 folds require `centroid_sim >= merge_sim` *in addition to* acceptance; FR-6 residual merges require `MergeSweeper.accepts_into`. Neither alone suffices: **the fold guard inspects the centroid but a fold transfers the REF_SET**, so a centroid-legitimate candidate carrying a contaminated ref_set poisons an LTM concept anyway (measured: `ltm_006` began with the tightest τ of all eight, 0.1387, and ran away to 0.7006 — cohesion 0.29 — capturing 169 known arrivals from seven other classes; with routing alone and no hooks it stays clean for all 2,000 steps, so the corruption is entirely hook-borne). Unguarded, T10 merged different-class candidates at centroid similarity down to **−0.443**: the residual pool is a pool of OUTLIERS, and density clustering over outliers groups them *as outliers, not as classes* — a shared HDBSCAN cluster is not evidence that two candidates are the same concept.
  4. **Merge admission is a DISJUNCTION, because `merge_sim` and τ measure different scales.** `merge_sim` is a **centroid-of-many** bar and only means what it says there (real data: same-class 20-sample half-centroids 0.94–0.98, cross-class 0.16–0.51). On **singletons** it is a category error — a same-class *single* embedding scores only ~0.735 against its own class centroid, so a 0.8 bar refuses ~68% of genuine same-class singleton merges. τ is the singleton-scale instrument (it *is* a per-concept calibrated acceptance radius for single embeddings) — but a seeded singleton's τ **is** the global prior (0.1537 ⇒ would demand sim ≥ 0.846, stricter still). So `accepts_into` admits iff **a host with a calibrated τ (ref_set ≥ 2) accepts the other's centroid, OR their centroids clear `merge_sim`**. Incoherent merges fail both legs; genuine consolidation passes at one scale or the other. **Never reuse `merge_sim` as a singleton bar, and never assume a singleton's τ is informative.**
- **Watch out: `min_cohesion` is calibrated for the synthetic worlds, not real data.** The fixture classes sit near 0.81, but real near-OOD class cohesion is **0.49–0.70** — straddling the §8 default `0.55`. Some real classes would fail FR-7's cohesion criterion (and, since T11, tier-1 eligibility) and could never promote. No synthetic test can surface this. It is an open owner calibration decision flagged for T13/T16 — do not silently "fix" it by editing the default.
- Everything is CPU-only. If a step is slow, optimize (vectorize) rather than reaching for GPU.
